# Лекция 11. Асинхронность в JavaScript: callback, Promise, async/await и event loop

![](./images/async_and_API.png)

## Введение

В прошлых лекциях мы много работали с тем, что происходит *“здесь и сейчас”*:
`нажали кнопку -> обработчик сработал -> DOM изменился -> пользователь увидел результат`.

Но в реальных приложениях очень быстро появляются задачи, которые не могут завершиться мгновенно:

- нужно подождать таймер;
- нужно дождаться ответа от браузерного API;
- нужно выполнить несколько шагов один за другим;
- нужно не дать пользователю сломать интерфейс повторными кликами;
- нужно показать состояние загрузки и корректно обработать ошибку.

Именно здесь начинается тема **асинхронности**.

Сегодня наша задача не *“ходить в настоящий API”*, а понять саму механику:

- чем асинхронный код отличается от синхронного;
- как браузер не блокирует интерфейс;
- как работают `callback`, `Promise`, `async/await`;
- как строить последовательные и параллельные асинхронные шаги;
- как правильно управлять состоянием интерфейса во время ожидания.

> **Важно:** эту лекцию мы сознательно держим сфокусированной на асинхронности.  
> В следующей теме эти же идеи мы перенесём на `fetch`, `HTTP` и реальный `API`.

## Кратко

Чтобы понять асинхронность, важно держать в голове несколько базовых мыслей:

- JavaScript выполняет код по одной инструкции за раз;
- долгие операции нельзя делать блокирующими, иначе интерфейс “зависает”;
- таймеры, события и сеть не крутятся “внутри JavaScript сами по себе” - браузер помогает ему через `Web APIs`;
- когда асинхронная операция заканчивается, её обработчик попадает в очередь и выполняется тогда, когда стек свободен;
- `callback`, `Promise` и `async/await` - это не три несвязанных темы, а три способа описывать ожидание результата.

Если коротко:

> **Асинхронность нужна для того, чтобы ждать результат, не блокируя остальной код и интерфейс.**

## Синхронный код и его проблемы

**Синхронный код** выполняется строго по порядку: строка за строкой. Следующая операция начнётся только тогда, когда закончится предыдущая.

Простой пример:

```javascript
console.log("1");
console.log("2");
console.log("3");
```

Результат всегда будет:

```text
1
2
3
```

Это удобно, потому что код предсказуем. Но у синхронного подхода есть важное ограничение:

> **Если одна операция занимает много времени, весь остальной код ждёт.**

А в браузере это особенно опасно, потому что браузер отвечает не только за вычисления, но и за:

- клики;
- скролл;
- ввод в `input`;
- анимации;
- перерисовку интерфейса.

Если главный поток заблокирован, пользователь воспринимает это как:

> **“Сайт завис”**

## Асинхронный код

**Асинхронный код** устроен иначе:

- мы запускаем операцию, которая завершится позже;
- не блокируем интерфейс в ожидании;
- продолжаем выполнять остальной код;
- а когда результат будет готов, выполняем заранее описанную логику.

Пример с таймером:

```javascript
console.log("Начало");

setTimeout(() => {
  console.log("Это выполнится через 1 секунду");
}, 1000);

console.log("Конец");
```

Результат:

```text
Начало
Конец
Это выполнится через 1 секунду
```

Ключевая мысль:

- `setTimeout` запускает таймер;
- но не останавливает выполнение программы;
- поэтому код после него выполняется сразу.

## Стек вызовов

Чтобы понять, почему это вообще работает, нужно знать, как JavaScript выполняет обычные функции.

**Стек вызовов (`Call Stack`)** - это место, где JavaScript хранит текущие вызовы функций.

Удобная модель:

- вызвали функцию;
- она попала в стек;
- внутри неё вызвали другую функцию;
- новая функция попала сверху;
- когда внутренняя функция завершилась, она ушла из стека;
- потом выполнение вернулось к предыдущей.

### Простой пример стека

```javascript
function outer() {
  function inner() {
    console.log("Hello!");
  }

  inner();
}

outer();
```

Что здесь происходит:

1. Вызвали `outer()` - она попала в стек.
2. Внутри вызвали `inner()` - она попала сверху.
3. Внутри `inner()` вызвали `console.log(...)`.
4. `console.log(...)` выполнился и ушёл из стека.
5. `inner()` завершилась и ушла из стека.
6. `outer()` завершилась и ушла из стека.

В синхронном коде всё именно так и работает: вызовы идут по цепочке, пока стек не опустеет.

## Цикл событий

Но с асинхронным кодом появляется новая задача:

> **Как выполнить что-то позже, если JavaScript в каждый момент делает только одно действие?**

Здесь уже нужен не только стек вызовов, но и среда браузера вокруг него.

В упрощённой модели есть 4 части:

1. **Call Stack** - где выполняется текущий код.
2. **Web APIs** - таймеры, события, сеть и другие механизмы браузера.
3. **Task Queue** - очередь, куда попадают готовые обработчики.
4. **Event Loop** - механизм, который следит, когда можно взять задачу из очереди и отдать её в стек.

### Web APIs

Когда вы вызываете:

- `setTimeout`
- обработчик события
- `fetch`

JavaScript не начинает сам “считать секунды” или “слушать браузерную кнопку”. Он передаёт эту работу браузеру.

Именно браузерные механизмы (`Web APIs`) занимаются:

- таймерами;
- обработкой кликов;
- сетью;
- многими другими внешними вещами.

### Task Queue

Когда асинхронная операция заканчивается, её callback не выполняется мгновенно. Сначала он попадает в **очередь задач**.

### Event Loop

**Event Loop** следит:

- пуст ли стек;
- есть ли готовые задачи в очереди.

Если стек свободен, `Event Loop` берёт следующую задачу из очереди и запускает её.

### Упрощённая схема

```text
JS code -> Call Stack
setTimeout / events / fetch -> Web APIs
callback after completion -> Task Queue
Event Loop -> переносит callback в Call Stack, когда стек свободен
```

### Пошаговый пример с `setTimeout`

```javascript
function main() {
  setTimeout(() => {
    console.log("Hello!");
  }, 2000);

  console.log("Bye!");
}

main();
```

Порядок такой:

1. `main()` попадает в стек.
2. Внутри вызывается `setTimeout(...)`.
3. Браузер регистрирует таймер во `Web APIs`, а сам `setTimeout` заканчивает работу и уходит из стека.
4. Выполняется `console.log("Bye!")`.
5. `main()` завершается, стек становится пустым.
6. Проходит 2 секунды.
7. Callback таймера попадает в `Task Queue`.
8. `Event Loop` видит, что стек пуст, и переносит callback в стек.
9. Выполняется `console.log("Hello!")`.

Из-за этого вывод будет таким:

```text
Bye!
Hello!
```

### Почему `setTimeout(..., 0)` не выполняется “мгновенно”

```javascript
console.log("A");

setTimeout(() => {
  console.log("B");
}, 0);

console.log("C");
```

Результат:

```text
A
C
B
```

Почему так:

1. `console.log("A")` выполняется сразу.
2. `setTimeout(..., 0)` только регистрирует таймер.
3. `console.log("C")` выполняется сразу.
4. Только после этого, когда стек освобождается, callback попадает в выполнение.

> **`0 ms` не означает “прямо сейчас”.**  
> Это означает: “выполни callback не раньше, чем текущий стек освободится”.

## Микрозадачи и макрозадачи: базовый уровень

Для сильного понимания асинхронности полезно знать ещё одну вещь:

- callbacks от `Promise` попадают в **микрозадачи**;
- callbacks от `setTimeout` попадают в **макрозадачи**.

Упрощённое правило:

1. Выполняется обычный синхронный код.
2. Когда стек пуст, сначала выполняются **микрозадачи**.
3. Потом берётся следующая **макрозадача**.

### Мини-пример

```javascript
console.log("1");

setTimeout(() => {
  console.log("timeout");
}, 0);

Promise.resolve().then(() => {
  console.log("promise");
});

console.log("2");
```

Результат будет таким:

```text
1
2
promise
timeout
```

Почему:

- `1` и `2` - это обычный синхронный код;
- `.then(...)` у `Promise` идёт в микрозадачи;
- `setTimeout` идёт в макрозадачи;
- микрозадача выполняется раньше таймера.

> На старте этого понимания достаточно.  
> Не нужно превращать первую лекцию по асинхронности в академический разбор event loop до атомов.

## Колбэки

Когда результат приходит позже, JavaScript должен как-то понять:

- что делать после завершения операции;
- когда именно выполнять этот код.

Самый старый и базовый механизм для этого - **callback**.

### Что такое callback

**Callback** - это функция, которую мы передаём в другую функцию, чтобы она была вызвана позже.

Мы буквально говорим:

> “Когда закончится это действие - вызови вот эту функцию.”

### Callback на примере событий

```javascript
button.addEventListener("click", function () {
  console.log("Кнопка была нажата");
});
```

Здесь переданная функция вызывается не сейчас, а только при событии `click`.

### Callback на примере таймера

```javascript
console.log("Старт");

setTimeout(() => {
  console.log("Прошла 1 секунда");
}, 1000);

console.log("Код идёт дальше");
```

### Callback как обычный приём

Callbacks используются не только для асинхронности. Это просто функция, переданная в другую функцию.

```javascript
function runLater(action) {
  console.log("Сейчас подготовка");
  action();
}

runLater(() => {
  console.log("Действие выполнено");
});
```

Но именно в асинхронном коде callbacks особенно важны, потому что помогают описывать логику “после завершения”.

## Проблема callback hell

Пока логика простая, callback - нормальный инструмент.

Например:

- обработать один клик;
- подождать один таймер;
- выполнить одно действие после завершения.

Но как только шагов становится много, вложенность быстро начинает расти.

### Пример

```javascript
setTimeout(() => {
  console.log("Шаг 1");

  setTimeout(() => {
    console.log("Шаг 2");

    setTimeout(() => {
      console.log("Шаг 3");
    }, 1000);
  }, 1000);
}, 1000);
```

Проблемы такого подхода:

- код уходит вправо;
- сложно читать;
- сложно добавлять обработку ошибок;
- сложно менять порядок шагов;
- сложно переиспользовать части логики.

Именно отсюда появился термин **callback hell**.

> Проблема не в том, что callback “плохой”.  
> Проблема в том, что он плохо масштабируется для длинных асинхронных цепочек.

## Промисы (`Promise`)

![](./images/promise.png)

Чтобы решить проблемы вложенности и разрозненной обработки ошибок, в JavaScript появился `Promise`.

### Что такое Promise

**Promise** - это объект, который представляет результат асинхронной операции.

У него есть три состояния:

- `pending` - ожидание;
- `fulfilled` - успешно завершён;
- `rejected` - завершён с ошибкой.

Главная идея:

- операция стартует сейчас;
- результат появится позже;
- `Promise` хранит это обещание будущего результата.

### Как создать Promise

```javascript
const promise = new Promise((resolve, reject) => {
  setTimeout(() => {
    resolve("Данные готовы");
    // reject(new Error("Ошибка загрузки"));
  }, 1000);
});
```

Здесь:

- `resolve(...)` переводит `Promise` в успешное состояние;
- `reject(...)` переводит `Promise` в ошибку.

### then: обработка успешного результата

```javascript
const promise = new Promise((resolve) => {
  setTimeout(() => resolve("Готово"), 1000);
});

promise.then((result) => {
  console.log("Результат:", result);
});
```

Смысл:

> “Когда `Promise` успешно завершится - выполни этот обработчик.”

### catch: обработка ошибки

```javascript
const promise = new Promise((resolve, reject) => {
  setTimeout(() => reject(new Error("Что-то пошло не так")), 1000);
});

promise
  .then((result) => {
    console.log("Успех:", result);
  })
  .catch((error) => {
    console.log("Ошибка:", error.message);
  });
```

### finally: код, который выполняется в любом случае

```javascript
const promise = new Promise((resolve) => {
  setTimeout(() => resolve("Готово"), 1000);
});

promise
  .then((result) => {
    console.log(result);
  })
  .catch((error) => {
    console.log(error.message);
  })
  .finally(() => {
    console.log("Операция завершилась");
  });
```

`finally` полезен для действий вроде:

- убрать `loading`;
- разблокировать кнопку;
- остановить индикатор;
- закрыть общий ресурс.

## Почему Promise удобнее callback

Главное преимущество `Promise` - **цепочки**.

```javascript
new Promise((resolve) => {
  setTimeout(() => resolve(1), 500);
})
  .then((value) => {
    console.log("Шаг 1:", value);
    return value + 1;
  })
  .then((value) => {
    console.log("Шаг 2:", value);
    return value + 1;
  })
  .then((value) => {
    console.log("Шаг 3:", value);
  })
  .catch((error) => {
    console.log("Ошибка:", error.message);
  });
```

Плюсы:

- код читается сверху вниз;
- шаги не уходят во вложенность;
- ошибки можно собирать в одном месте;
- каждый шаг может вернуть новый результат дальше по цепочке.

## Важный момент: `then` должен что-то возвращать

Одна из самых частых ошибок новичков - забыть вернуть значение из `then`.

### Правильно

```javascript
Promise.resolve(10)
  .then((value) => {
    return value * 2;
  })
  .then((value) => {
    console.log(value); // 20
  });
```

### Ошибка

```javascript
Promise.resolve(10)
  .then((value) => {
    value * 2; // ничего не вернули
  })
  .then((value) => {
    console.log(value); // undefined
  });
```

Если в `then` ничего не вернуть, следующий шаг получит `undefined`.

## Как писать свои асинхронные функции

На практике очень полезно иметь маленькие функции, которые возвращают `Promise`.

Например, классическая утилита `wait`:

```javascript
function wait(ms) {
  return new Promise((resolve) => {
    setTimeout(resolve, ms);
  });
}
```

Теперь мы можем строить последовательность шагов:

```javascript
wait(1000)
  .then(() => {
    console.log("Прошла 1 секунда");
    return wait(1000);
  })
  .then(() => {
    console.log("Прошла ещё 1 секунда");
  });
```

Или вернуть данные:

```javascript
function fakeRequest(name, ms) {
  return new Promise((resolve) => {
    setTimeout(() => {
      resolve(`${name} готов`);
    }, ms);
  });
}

fakeRequest("Профиль", 1200).then((message) => {
  console.log(message);
});
```

## Асинхронные функции

`async/await` - это не “другой механизм вместо Promise”.

Это более удобная запись работы с `Promise`.

### `async`: функция всегда возвращает `Promise`

```javascript
async function getMessage() {
  return "Привет";
}

console.log(getMessage()); // Promise
```

Даже если внутри обычный `return`, наружу всё равно возвращается `Promise`.

### Связка `async/await`

```javascript
function wait(ms) {
  return new Promise((resolve) => {
    setTimeout(resolve, ms);
  });
}

async function run() {
  console.log("Начало");

  await wait(1000);
  console.log("Прошла 1 секунда");

  await wait(1000);
  console.log("Прошла ещё 1 секунда");

  console.log("Конец");
}

run();
```

Код читается почти как синхронный:

- дошли до `await`;
- дождались результата;
- пошли дальше.

### Плюсы `async/await`

Частая ошибка:

> “Если я написал `await`, значит браузер остановился.”

Это не так.

`await` делает паузу только внутри текущей `async`-функции. При этом:

- интерфейс продолжает жить;
- клики работают;
- анимации идут;
- другие задачи могут выполняться.

### Как переписать цепочку `.then()` в `async/await`

### Вариант через `then`

```javascript
wait(1000)
  .then(() => {
    console.log("Шаг 1");
    return wait(1000);
  })
  .then(() => {
    console.log("Шаг 2");
    return wait(1000);
  })
  .then(() => {
    console.log("Шаг 3");
  });
```

### То же самое через `async/await`

```javascript
async function runSteps() {
  await wait(1000);
  console.log("Шаг 1");

  await wait(1000);
  console.log("Шаг 2");

  await wait(1000);
  console.log("Шаг 3");
}

runSteps();
```

Когда шагов много, второй вариант обычно читается легче.

## Обработка ошибок в асинхронном коде

С асинхронностью всегда есть два сценария:

- всё прошло успешно;
- что-то пошло не так.

Если ошибки не обрабатывать, интерфейс становится непредсказуемым:

- кнопки могут остаться заблокированными;
- статус может так и остаться `Loading...`;
- пользователь не поймёт, что случилось.

### throw: как явно создать ошибку

```javascript
function checkNumber(value) {
  if (typeof value !== "number") {
    throw new Error("Ожидалось число");
  }

  return value * 2;
}

try {
  console.log(checkNumber("10"));
} catch (error) {
  console.log("Ошибка:", error.message);
}
```

`throw` нужен, когда вы хотите явно прервать выполнение и сказать:

> “Дальше идти нельзя, состояние некорректное.”

### catch в Promise-цепочке

```javascript
function failAfter(ms) {
  return new Promise((resolve, reject) => {
    setTimeout(() => {
      reject(new Error("Что-то пошло не так"));
    }, ms);
  });
}

failAfter(1000)
  .then(() => {
    console.log("Успех");
  })
  .catch((error) => {
    console.log("Поймали ошибку:", error.message);
  });
```

### try/catch в async/await

```javascript
async function run() {
  try {
    console.log("Старт");
    await failAfter(1000);
    console.log("Этот код не выполнится");
  } catch (error) {
    console.log("Поймали ошибку:", error.message);
  }
}

run();
```

С `async/await` именно `try/catch` обычно становится основным способом обработки ошибок.

### finally: завершение независимо от результата

```javascript
function wait(ms) {
  return new Promise((resolve) => setTimeout(resolve, ms));
}

async function load() {
  try {
    console.log("Показываем loading...");
    await wait(1000);
    console.log("Получили данные");
  } catch (error) {
    console.log("Ошибка:", error.message);
  } finally {
    console.log("Убираем loading в любом случае");
  }
}

load();
```

`finally` особенно важен в UI-логике.

## Важная ловушка: `try/catch` не ловит ошибки из будущего callback

Вот пример, который часто удивляет новичков:

```javascript
try {
  setTimeout(() => {
    throw new Error("Ошибка внутри таймера");
  }, 0);
} catch (error) {
  console.log("Не сработает:", error.message);
}
```

Почему `catch` не сработал:

- `try/catch` завершился сразу;
- callback таймера выполнился позже;
- это уже другой момент выполнения.

Если хотите управляемо обрабатывать такие сценарии, нужно работать через `Promise` и `reject`, а не надеяться, что внешний `try/catch` поймает будущую ошибку.

## Последовательное и параллельное выполнение

Асинхронный код можно строить по-разному:

- **последовательно** - когда второй шаг зависит от первого;
- **параллельно** - когда шаги независимы и их можно запускать одновременно.

### Последовательно

```javascript
async function runSequential() {
  const user = await fakeRequest("user", 1000);
  const settings = await fakeRequest("settings", 800);

  console.log(user);
  console.log(settings);
}
```

Этот подход нужен, если:

- второй шаг нельзя выполнить без результата первого;
- есть строгая бизнес-логика;
- действия должны идти в конкретном порядке.

### Параллельно через `Promise.all`

Если шаги независимы, лучше запускать их одновременно:

```javascript
async function runParallel() {
  const [user, settings] = await Promise.all([
    fakeRequest("user", 1000),
    fakeRequest("settings", 800),
  ]);

  console.log(user);
  console.log(settings);
}
```

Здесь оба запроса стартуют сразу. Это быстрее, чем ждать их по очереди.

### Когда использовать `Promise.all`

Используйте `Promise.all`, если:

- задачи независимы;
- нужен результат всех задач;
- если одна задача упала, весь сценарий должен считаться ошибкой.

> `Promise.all` “падает” сразу, если хотя бы один `Promise` завершился с ошибкой.

## На практике

Теперь соберём один более цельный пример, как это выглядит в интерфейсе.

### Практический паттерн: асинхронность в интерфейсе

В реальном интерфейсе асинхронность почти всегда означает одну и ту же задачу:

1. Пользователь запускает действие.
2. Мы показываем `loading`.
3. Блокируем повторный запуск.
4. Ждём результат.
5. Показываем успех или ошибку.
6. Возвращаем интерфейс в нормальное состояние.

Это можно тренировать и без настоящего `API`, используя имитацию асинхронной операции.

### Базовая функция `fakeRequest`

```javascript
function fakeRequest(label, ms, shouldFail = false, data = null) {
  return new Promise((resolve, reject) => {
    setTimeout(() => {
      if (shouldFail) {
        reject(new Error(`${label}: ошибка`));
        return;
      }

      resolve(data ?? `${label}: готово`);
    }, ms);
  });
}
```

Она делает три важные вещи:

- ждёт `ms` миллисекунд;
- может завершиться успехом;
- может завершиться ошибкой.

То есть это отличный учебный заменитель реального сервера.

## Мини-проект внутри лекции: симуляция загрузки дашборда

Сейчас соберём маленький пример, который очень похож на реальную frontend-задачу.

Идея:

- пользователь нажимает кнопку `Load dashboard`;
- сначала идёт “авторизация”;
- потом в параллель грузятся `profile`, `courses`, `notifications`;
- во время загрузки интерфейс показывает статус;
- при ошибке пользователь получает понятное сообщение;
- повторный клик во время загрузки блокируется.

### Разметка

```html
<button id="loadBtn">Load dashboard</button>
<button id="failBtn">Load with error</button>

<p id="status"></p>
<div id="app"></div>
```

### Подготовка DOM-элементов

```javascript
const loadBtn = document.getElementById("loadBtn");
const failBtn = document.getElementById("failBtn");
const statusEl = document.getElementById("status");
const appEl = document.getElementById("app");

function setStatus(text) {
  statusEl.textContent = text;
}

function clearApp() {
  appEl.innerHTML = "";
}

function setLoadingState(isLoading) {
  loadBtn.disabled = isLoading;
  failBtn.disabled = isLoading;
}
```

### Имитация асинхронных шагов

```javascript
function authorizeUser() {
  return fakeRequest("auth", 800, false, { userId: 7, name: "Alex" });
}

function loadProfile(userId) {
  return fakeRequest("profile", 1000, false, {
    userId,
    city: "Prague",
    level: "junior",
  });
}

function loadCourses(userId) {
  return fakeRequest("courses", 1200, false, [
    { id: 1, title: "JavaScript Basics" },
    { id: 2, title: "DOM Practice" },
  ]);
}

function loadNotifications(userId, shouldFail = false) {
  return fakeRequest("notifications", 700, shouldFail, [
    "New lesson available",
    "Homework deadline tomorrow",
  ]);
}
```

### Рендер результата

```javascript
function renderDashboard(user, profile, courses, notifications) {
  appEl.innerHTML = `
    <h2>${user.name}</h2>
    <p>City: ${profile.city}</p>
    <p>Level: ${profile.level}</p>
    <p>Courses: ${courses.length}</p>
    <p>Notifications: ${notifications.length}</p>
  `;
}
```

### Основная логика

```javascript
async function loadDashboard(shouldFail = false) {
  if (loadBtn.disabled || failBtn.disabled) return;

  setLoadingState(true);
  setStatus("Loading dashboard...");
  clearApp();

  try {
    const user = await authorizeUser();

    const [profile, courses, notifications] = await Promise.all([
      loadProfile(user.userId),
      loadCourses(user.userId),
      loadNotifications(user.userId, shouldFail),
    ]);

    renderDashboard(user, profile, courses, notifications);
    setStatus("Dashboard loaded");
  } catch (error) {
    setStatus(`Error: ${error.message}`);
  } finally {
    setLoadingState(false);
  }
}

loadBtn.addEventListener("click", () => loadDashboard(false));
failBtn.addEventListener("click", () => loadDashboard(true));
```

### Что здесь важно понять

Этот пример важен не из-за конкретных данных, а из-за паттерна:

1. В начале блокируем интерфейс.
2. Показываем `loading`.
3. Выполняем асинхронные шаги.
4. На успехе - рендерим результат.
5. На ошибке - показываем понятный статус.
6. В `finally` возвращаем кнопки в рабочее состояние.

Это тот же самый каркас, который позже будет использоваться с `fetch`.

## Частые ошибки новичков

### 1) Думать, что `setTimeout` блокирует код

Нет. `setTimeout` лишь регистрирует таймер. Код после него выполняется сразу.

### 2) Думать, что `await` замораживает страницу

Нет. `await` делает паузу только внутри текущей `async`-функции.

### 3) Забывать `return` в `.then()`

Тогда следующий шаг получает `undefined`.

### 4) Писать всё последовательно, даже если можно параллельно

Плохо:

```javascript
const a = await loadProfile();
const b = await loadCourses();
const c = await loadNotifications();
```

Если шаги независимы, это медленнее, чем нужно.

Лучше:

```javascript
const [a, b, c] = await Promise.all([
  loadProfile(),
  loadCourses(),
  loadNotifications(),
]);
```

### 5) Не использовать `finally` для восстановления UI

Если в `catch` забыли вернуть кнопку в рабочее состояние, интерфейс может застрять в “поломанном” виде.

### 6) Не защищать действие от повторного запуска

Если пользователь нажмёт кнопку 5 раз подряд, вы можете получить 5 асинхронных сценариев одновременно.

### 7) Ловить не ту ошибку и не там

Внешний `try/catch` не ловит ошибку из будущего callback таймера.

### 8) Смешивать бизнес-логику и UI без структуры

Если в одной функции сразу:

- логика данных;
- таймеры;
- `DOM`;
- статусы;
- обработка ошибок;

код быстро становится трудным для чтения. Даже в маленьких учебных проектах полезно разделять:

- асинхронные функции;
- функции рендера;
- обработчики событий.
